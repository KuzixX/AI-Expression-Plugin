# Heat-Align Transfer — описание метода

> Автор идеи: пользователь.
> Документация: 2026-05-25 (v4 base).

Метод **переноса motion-groups** (мышечно-кластерных деформаций) между мешами
**разных топологий и пропорций** через **корреспонденцию в пространстве тепловых
отпечатков**.

В одну строку: *тепло — это «отпечаток пальца» вершины; одинаковые отпечатки на
разных мешах = анатомически одна и та же точка → можно копировать
принадлежность к кластеру.*

---

## 1. Постановка задачи

**Дано:**
- Меш A (source, например FLAME): `V_A` вершин, `F_A` треугольников
- Меш B (target, например FBX): `V_B` вершин, `F_B` треугольников
- На A уже:
  - выбраны K **anchor-точек** (опорных вершин, типа "центр носа", "уголок рта")
  - вычислена **деформация** δ_A для какого-то выражения лица
  - δ_A **разбита на кластеры** (motion-groups) — каждая группа имеет свою
    локальную трансформацию `(R_c, μ_c, c_c)` (поворот, сдвиг, центр)
- На B выбраны **те же** K anchor-точек (по соответствию: "вот это нос на FLAME,
  вот это нос на FBX")

**Найти:** деформацию δ_B на меше B, которая воспроизводит то же выражение.

**Сложность:** топологии разные. У FLAME 5023 вершины, у FBX может быть 8000+,
и они расположены **в разных местах**. Vertex i в FLAME ≠ vertex i в FBX.

---

## 2. Существующие подходы и их недостатки

| Подход | Идея | Проблема |
|---|---|---|
| **Nearest xyz** | Для каждой B-вершины найти ближайшую по координатам A-вершину | Ломается на разных пропорциях (нос FBX смотрит на щёку FLAME) |
| **Voronoi/Dijkstra** | Залить B-меш от центроидов A-кластеров через геодезию | Центроид кластера ≠ анатомическая точка; для разных пропорций центроид «уплывает» |
| **HKS / WKS** | Спектральные intrinsic-дескрипторы (eigendecomp Laplace-Beltrami) | Сильно зависят от полной геометрии; симметричные точки путаются; дорогие |
| **Functional Maps** | Линейные отображения между спектральными базами | Тяжёлая теория; требует много corresponding landmarks |

Все они **не используют главный имеющийся сигнал — heat-диффузию от anchor-точек**,
которая по построению уже даёт информацию о том «где относительно anchor'ов
находится каждая вершина».

---

## 3. Идея метода — подробный разбор

Метод состоит из **5 шагов**: heat-диффузия → нормализация → корреспонденция →
наследование label → применение трансформации. Ниже каждый шаг разобран
с математикой, кодом и объяснением «почему именно так».

---

### 3.1 Heat как «отпечаток вершины»

#### Что вычисляем

Для каждого anchor-узла `a ∈ {0..K-1}` решаем уравнение теплопроводности
на меше:

```
∂u/∂t = Δu     с начальным условием u(v, 0) = δ_{v, a}
```

где `Δ` — **Laplace-Beltrami оператор** на триангулированной поверхности.
Берём фиксированный момент времени `t > 0` (параметр алгоритма):

```
u_a(v, t) = (exp(t·Δ) · δ_a) [v]
```

Это **тепло в вершине v** после диффузии из точечного источника в `a` за время `t`.

#### Дискретизация: cotangent Laplacian + implicit Euler

На меше `Δ` строится через котангенсный оператор:

```
L_{ij} = (cot α_{ij} + cot β_{ij}) / 2   для рёбер (i,j)
L_{ii} = -Σ_{j} L_{ij}
```

где `α, β` — углы напротив ребра в двух соседних треугольниках.
**Mass matrix** `M` = диагональная, `M_{ii}` = `1/3` × сумма площадей треугольников
вокруг вершины `i` (Voronoi-area вариант).

Решаем **implicit Euler** одним шагом за `t`:

```
(M - t·L) · u(t) = M · u(0)
```

Это разреженная SPD-система → решаем `scipy.sparse.linalg.spsolve` за O(N√N).

#### Mass-normalized initial condition

В коде:

```python
u0 = np.zeros(N)
u0[anchor_idx] = 1.0 / M[anchor_idx, anchor_idx]   # делим на свою mass-area
```

Зачем делим на `M[anchor, anchor]`? Чтобы **полная масса тепла** была одна
независимо от того в какую вершину поставили источник (густой район vs
разреженный). Это первая нормализация — на уровне initial condition.

#### Собираем матрицу

После K решений системы (по одному на anchor) имеем:

```
H_A ∈ ℝ^(K × N_A)    H_A[a, v] = u_a(v, t)  на меше A
H_B ∈ ℝ^(K × N_B)    H_B[a, v] = u_a(v, t)  на меше B
```

Транспонируя — каждая **колонка** это per-vertex K-мерный вектор:

```
h_A(v) = (H_A[0,v], H_A[1,v], ..., H_A[K-1,v])  ∈ ℝ^K
h_B(u) = (H_B[0,u], H_B[1,u], ..., H_B[K-1,u])  ∈ ℝ^K
```

#### Почему это «отпечаток»

Вектор `h(v)` — это **K чисел**, каждое из которых говорит «насколько эта
вершина близка (по геодезии) к anchor-у `a`». Близко = высокое значение,
далеко = почти ноль (тепло экспоненциально затухает с расстоянием по правилу
`u ≈ (4πt)^{-1/2} · exp(-d²/4t)`).

**Геометрическая интуиция:** K анкеров задают K-мерное «координатное
пространство» на меше. Каждая вершина имеет в нём свой «адрес» — типа
«я возле носа, далеко от лба, среднее расстояние от рта». На двух мешах
точка которая семантически «кончик носа» должна иметь практически
одинаковый адрес, даже если её xyz совершенно разные.

#### Почему именно heat (а не геодезия напрямую)

Можно было бы вместо тепла брать чистую геодезическую дистанцию. Но:
- Геодезия дороже считать (Fast Marching / Dijkstra на каждой вершине)
- Геодезия — **negotiated** (resh-sensitive): один сильно-неравномерный
  триангль может сильно повлиять на дистанцию через него
- Heat — **усреднённая** дистанция по множеству путей, гладкая, устойчивая
  к местным дефектам триангуляции
- Heat имеет **естественный параметр `t`** — позволяет регулировать масштаб
  «видимости» от каждого anchor'а

#### Что значит выбор параметра `t`

- Маленькое `t` → тепло остаётся возле anchor'а → каждая вершина видит
  только ближайшие 1-2 anchor'а → отпечатки бедные, плохо различающие
- Большое `t` → тепло размазалось по всему мешу → все вершины имеют
  похожие отпечатки → разрешающая способность падает
- **Правильное `t`** — такое чтобы зоны анкеров слегка перекрывались;
  обычно подбирается так что максимальная heat-зона покрывает ~30-60%
  меша

### 3.2 Нормализация (критический шаг)

#### Проблема

После heat-диффузии на меше с **N=5000 вершин** и `M ≈ 0.5` per vertex
максимальное значение heat ≈ `2.0`. На меше с **N=18000 вершин** и
`M ≈ 0.15` per vertex максимум будет ≈ `6.5`. Разные scales → прямой
nearest neighbor сравнит `0.5` с `4.5` и решит «они далеко».

Реальный пример из дампа:
```
HEAD1 (5023 верт): max(heat) per anchor ≈ [4.8, 5.2, 3.9, ...]
FBX   (7799 верт): max(heat) per anchor ≈ [12.3, 14.1, 9.8, ...]
```

В **3 раза** разные. Без нормализации алгоритм сломан.

#### Что делаем: per-anchor max-normalize

Для каждого anchor отдельно:

```python
H_A_norm[a, :] = H_A[a, :] / H_A[a, :].max()    # → диапазон [0, 1]
H_B_norm[a, :] = H_B[a, :] / H_B[a, :].max()
```

Теперь обе матрицы имеют один и тот же scale `[0, 1]` независимо от плотности
сетки.

#### Почему именно per-anchor (а не глобально)

Альтернативы и почему они хуже:

| Способ | Что плохо |
|---|---|
| Glob max (`H / H.max()`) | Один доминирующий anchor забивает все остальные; если у anchor 0 max=10, а у anchor 5 max=0.5, то после норм. anchor 5 становится почти `0` для всех вершин |
| L2 per vertex (`h(v) / ||h(v)||`) | Теряется абсолютная информация — две вершины с очень разной общей теплоёмкостью становятся неотличимы |
| Z-score (`(h - mean) / std`) | Уводит heat в отрицательные значения, нарушает физическую интерпретацию |
| Min-max per anchor | Эквивалентно нашему варианту т.к. min ≈ 0 у heat (тепло не отрицательное) |

**Per-anchor max-normalize** сохраняет:
- Относительное распределение тепла внутри одного anchor'а
- Сравнимость между мешами
- Физический смысл (`1.0` = «горячее место»)

#### Геометрическая интерпретация

После нормализации `h_norm(v)` это **«профиль вершины v в anchor-coordinates»**:
- `0.8` для anchor `a` → «эта вершина в горячей зоне `a`»
- `0.05` для anchor `b` → «эта вершина на периферии зоны `b`»

Этот профиль **топология-инвариантен**: одна анатомическая точка на двух
мешах даёт почти одинаковые профили.

### 3.3 Корреспонденция вершин

#### Постановка

Для каждой target-вершины `u ∈ V_B` найти source-вершину `v* ∈ V_A` с
**максимально похожим heat-отпечатком**:

```
v*(u) = argmin_{v ∈ V_A}  || h_A_norm(v) - h_B_norm(u) ||²
```

#### Метрика расстояния

Используется **squared L2** в K-мерном пространстве. Эквивалентно
**cosine distance** после нормализации (потому что обе нормированы и в
позитивном орт-конусе) — кому какая интуиция ближе.

В коде через identity `||a-b||² = ||a||² + ||b||² - 2⟨a,b⟩`:

```python
a_sq = (H_t_zone ** 2).sum(1, keepdims=True)      # (M, 1)
b_sq = (H_s_zone ** 2).sum(1, keepdims=True).T    # (1, S)
cross = H_t_zone @ H_s_zone.T                     # (M, S)
D2 = a_sq + b_sq - 2 * cross                      # (M, S)
```

Это **один матричный продукт** вместо двойного цикла — O(M·S·K) operations,
которые numpy / BLAS выполняют за миллисекунды.

#### Ограничение зоной anchor'а

Если искать корреспонденцию по всему мешу, могут произойти патологические
сопоставления — например, если у вершины на левой щеке профиль случайно
оказался похож на правую (симметрия лица).

Решение: для каждого anchor `a` ищем корреспонденцию **только среди вершин
этого anchor'а**:

```python
for a in range(K):
    active_target = {u : H_B_norm[a, u] > threshold}       # M вершин
    active_source = {v : v ∈ кластере anchor'а a}         # S вершин

    # ищем корреспонденцию только M × S, не M × N_A
    correspondence = nn_match(active_target, active_source)
```

Это **зональная декомпозиция** — каждый anchor обрабатывается изолированно,
ошибки не «протекают» между зонами.

`threshold` обычно 5% от максимума (вершина считается «принадлежащей» зоне
если её heat > 0.05 от max).

#### k-NN (вместо одного nearest)

Реальная реализация берёт **top-k** ближайших source-вершин, а не одну —
это даёт устойчивость к шуму границ. Подробнее в §4.1.

#### Сложность

- M активных target, S кластеризованных source per anchor
- На anchor: O(M · S · K) = ~10000 × 1000 × 6 = 60M операций → <100ms в numpy
- На все K анкеров: ~K · 60M = 360M → <1 сек

Это **на порядки быстрее** Functional Maps (которые требуют SVD большой
матрицы и iterative refinement).

#### Почему это работает

Гипотеза метода (которая эмпирически подтверждается): **функция
`h_norm: V → [0,1]^K` инъективна на анатомически осмысленных точках, и
её композиция `h_B_norm ∘ φ ≈ h_A_norm` для правильной корреспонденции `φ`.**

Другими словами, K-мерный heat-profile — это **достаточно богатая
сигнатура чтобы различать анатомические точки**, и **достаточно
интрисик чтобы быть устойчивой между мешами**.

Минимальное K зависит от сложности анатомии. Для лица обычно достаточно
4-8 анкеров.

### 3.4 Наследование cluster-label

#### Reverse lookup

На меше A у нас есть кластеризация — каждой source-вершине известен её
кластер:

```python
v_to_cluster = {}                              # dict: vertex_id → cluster_obj
for cluster in clusters_A:
    for v in cluster.indices:
        v_to_cluster[v] = cluster
```

#### Перенос label через корреспонденцию

Target-вершина наследует label своей корреспонденции:

```python
for u in active_target:
    v_star = correspondence[u]                 # nearest source vertex
    target_label[u] = v_to_cluster[v_star]
```

#### Группировка → target_clusters

После цикла по всем target-вершинам имеем `dict {u: cluster_obj}`.
Группируем по cluster:

```python
target_clusters = defaultdict(list)
for u, cl_obj in target_label.items():
    target_clusters[cl_obj].append(u)

# теперь target_clusters[cl] = список target-вершин, унаследовавших cl
```

#### Формальная запись

Если обозначить:
- `φ: V_B → V_A` — функция корреспонденции
- `c_A: V_A → ℤ` — cluster label на source
- `c_B: V_B → ℤ` — индуцированный cluster label на target

то

```
c_B = c_A ∘ φ
```

Это **pullback** функции `c_A` через `φ`. В терминах Functional Maps это
обычный operator `C·f`, где `C` — discrete функциональная матрица
корреспонденции (binary permutation в нашем случае).

#### Почему это валидно

Cluster labels — это **дискретное поле** на меше A (типа окраски).
Если корреспонденция `φ` правильная, то pullback переносит окраску
**пиксель-в-пиксель** — каждая target-точка попадает в кластер своего
анатомического двойника.

Возможные артефакты:
- **Спекл-шум** — соседние target-вершины могут наследовать разные labels
  если корреспонденция не идеальна. Лечится k-NN voting (§4.1) и
  mesh-graph smoothing (§4.2).
- **Полное «съедание» кластера** — если ни одна target-вершина не
  выбирает данный source-кластер (его профиль слишком уникален и нет
  похожих на target), кластер просто пропадает на target. Обычно это
  ок — значит на target его и не должно быть.

#### Где это происходит в коде

```python
# debug_head1_pipeline.py, функция assign_target_to_source_by_heat_align
fbx_to_cluster = {}
for a, src_list in src_by_anchor.items():
    # ... вычисляем D2, выбираем nearest или top-k ...
    for t_v, s_v in zip(active_t_idx, nearest_src_global):
        cl_obj, _ = vertex_to_cluster.get(s_v, (None, None))
        if cl_obj is not None:
            fbx_to_cluster[t_v] = cl_obj
```

Структура `fbx_to_cluster` это и есть `c_B = c_A ∘ φ`.

### 3.5 Применение трансформации

#### Что у нас есть в source-кластере

Когда мы кластеризовали движение на меше A, для каждого кластера через
**polar decomposition** вычислили локальную трансформацию:

```
(R_c, μ_c, c_c, S_c)
```

- `R_c ∈ SO(3)` — ortogонал. матрица поворота
- `μ_c ∈ ℝ³` — средний сдвиг (translation)
- `c_c ∈ ℝ³` — центроид кластера в rest-позе (`weighted mean` по `heat_weights`)
- `S_c` — симметричная stretch-matrix (масштабирование вдоль 3 осей)

Эти величины вычисляются один раз на этапе анализа деформации δ_A.

#### Локальная аффинная трансформация

Для каждой вершины кластера применяем:

```
v_new = c_c + R_c · (v_rest - c_c) + μ_c
```

или эквивалентно:

```
δ(v) = R_c · (v_rest - c_c) + c_c + μ_c - v_rest
       = (R_c - I) · (v_rest - c_c) + μ_c
```

Геометрически:
1. **Сдвинули в центроид:** `v_rest - c_c` — поставили центр кластера в 0
2. **Повернули:** `R_c · (...)` — крутим всю группу вокруг центра
3. **Вернули обратно:** `+ c_c`
4. **Добавили общий сдвиг:** `+ μ_c`

#### Применение к target-мешу

То же самое, но для target-вершин с **тем же `(R_c, μ_c, c_c)` от source**:

```python
delta_B = np.zeros((N_B, 3))
for cl_obj, target_verts in target_clusters.items():
    R, mu, c = cl_obj['R'], cl_obj['mu'], cl_obj['c_rest']
    for u in target_verts:
        v_rest = verts_B[u]
        delta_B[u] = R @ (v_rest - c) + c + mu - v_rest
```

Заметь: `c_c` берётся **из source-кластера**, не пересчитывается для target.
Это потенциальная неточность — на разных мешах «центр» одного и того же
кластера лежит в немного разных местах. В текущей реализации мы принимаем
эту неточность как малую (т.к. кластеры локальны), но в v4 можно
пересчитывать центр для target отдельно.

#### Финальный Laplacian smoothing

Per-cluster трансформация = аффинная функция. На **границах кластеров**
получается **разрыв первой производной** (соседние вершины принадлежат
разным кластерам, имеют разные R/μ → их финальные позиции образуют ступеньку).

Сглаживаем итерациями mesh-Laplacian:

```
δ^{k+1}(v) = (1-α) · δ^k(v) + α · mean(δ^k(neighbors(v)))
```

Это **усреднение по 1-кольцу** с весом α (обычно 0.5), повторённое N раз
(обычно 30 для FBX). Каждая итерация распространяет сглаживание на 1 ring
дальше → за 30 итераций сглаживание распространяется на ~30 шагов соседства.

Параметры:
- `smooth_iters_fbx` — число итераций (default 30, авто-скейл по размеру)
- `smooth_alpha` — сила усреднения (default 0.5)

#### Почему сглаживается именно δ, а не v

Сглаживаем **дельты** (сдвиги от rest), не финальные позиции. Это важно:
- Если сглаживать `v_new`, мы потеряем форму rest-меша (загладим складки кожи)
- Если сглаживать `δ`, мы сглаживаем только **движение**, а rest-форма
  остаётся как есть

После сглаживания финальная позиция:

```
v_final = v_rest + smoothed_delta
```

#### Итоговая формула метода

Объединяя всё:

```
v_final(u) = v_B(u) + Smooth^N[ R_{c_A(φ(u))} · (v_B(u) - c_{c_A(φ(u))})
                                + c_{c_A(φ(u))} + μ_{c_A(φ(u))} - v_B(u) ](u)
```

Читается: «для каждой target-вершины `u` вычисляем дельту через source-кластер
её анатомического двойника `φ(u)`, потом сглаживаем дельту по mesh-графу
N раз, прибавляем к rest-позиции».

---

### Резюме шага 3

| Подшаг | Вход | Выход | Что делает |
|---|---|---|---|
| 3.1 Heat | mesh, anchors | `H ∈ ℝ^(K×N)` | Решает уравнение теплопроводности из K источников |
| 3.2 Normalize | `H_A`, `H_B` | `H_A_norm`, `H_B_norm` ∈ `[0,1]` | Per-anchor max → сравнимо между мешами |
| 3.3 Correspond | `H_A_norm`, `H_B_norm` | `φ: V_B → V_A` | Nearest neighbor в K-мерном heat-space, по зонам |
| 3.4 Inherit | `φ`, `c_A` | `c_B = c_A ∘ φ` | Pullback cluster-labels на target |
| 3.5 Transform | `c_B`, `(R, μ, c)_c` | `δ_B` | Аффинная трансформация per кластер + Laplacian smooth |

## 4. Регуляризация (антишумовая)

Чистый `argmin` может давать «спекл-шум» на границах: соседние target-вершины
выбирают разные source-вершины из разных кластеров и получают разные labels.
Две регуляризации:

### 4.1 k-NN weighted voting

Вместо одного `argmin` берём **top-k** ближайших source-вершин и голосуем
за их cluster label с весами:

```
weight(v_j) = 1 / (d²(h_A(v_j), h_B(u)) + ε)
cluster(u) = argmax_c  Σ_{v_j ∈ topK, c(v_j)=c}  weight(v_j)
```

`k_nn=1` → чистый argmin. `k_nn=5..10` → подавление одиночных перескоков.

### 4.2 Mesh-graph label smoothing

После назначения labels — пройдёмся по edge-графу B и для каждой вершины
поменяем label на **самый частый** среди 1-ring соседей (majority vote).
2-3 итерации убирают одиночные «островки» в чужом кластере.

---

## 5. Алгоритм (псевдокод)

```python
def heat_align_transfer(mesh_A, mesh_B, anchors_A, anchors_B,
                         clusters_A,  # с трансформациями (R, μ, c)
                         k_nn=5, smooth_iters=2):

    # 1. Heat-диффузия из каждого anchor отдельно
    H_A = [heat_diffuse(mesh_A, a) for a in anchors_A]  # shape (K, N_A)
    H_B = [heat_diffuse(mesh_B, a) for a in anchors_B]  # shape (K, N_B)

    # 2. Per-anchor max-normalization
    H_A /= H_A.max(axis=1, keepdims=True)
    H_B /= H_B.max(axis=1, keepdims=True)

    # 3. Reverse lookup: vertex_A → cluster
    v_to_cluster = {v: c for c in clusters_A for v in c.vertices}

    # 4. Для каждой active target-вершины ищем корреспонденцию
    target_labels = {}
    for anchor_idx in range(K):
        active_B = [u for u in mesh_B.verts
                     if H_B[anchor_idx, u] > threshold * H_B[anchor_idx].max()]
        active_A_clustered = [v for v in v_to_cluster
                               if v in zone_of(anchor_idx)]

        for u in active_B:
            # top-k ближайших по K-мерному heat-вектору
            distances = [|| H_A[:, v] - H_B[:, u] ||² for v in active_A_clustered]
            topk = sorted(zip(distances, active_A_clustered))[:k_nn]

            # взвешенное голосование за cluster
            votes = {}
            for d, v in topk:
                c = v_to_cluster[v]
                votes[c] = votes.get(c, 0) + 1 / (d + ε)
            target_labels[u] = argmax(votes)

    # 5. Mesh-graph smoothing labels (majority vote по соседям)
    for _ in range(smooth_iters):
        new_labels = {}
        for u in target_labels:
            counts = Counter([target_labels[n] for n in neighbors(u)])
            new_labels[u] = counts.most_common(1)[0][0]
        target_labels = new_labels

    # 6. Группируем + применяем трансформации source-кластеров
    delta_B = zeros((N_B, 3))
    for u, c in target_labels.items():
        delta_B[u] = c.R @ (mesh_B.verts[u] - c.center) + c.center + c.μ \
                      - mesh_B.verts[u]

    # 7. Laplacian smoothing на edge-графе B
    delta_B = laplacian_smooth(delta_B, mesh_B.faces, n_iter=30)

    return delta_B
```

## 6. Почему это работает

### 6.1 Heat — это устойчивая координата на поверхности

Heat-equation `∂u/∂t = Δu` с источником в anchor даёт **гладкую монотонно убывающую
функцию** по геодезическому расстоянию. Она не зависит от:
- ориентации и масштаба меша (intrinsic)
- частоты дискретизации (после max-нормализации)
- мелких локальных деталей геометрии (диффузия их сглаживает)

### 6.2 K анкеров → K-мерная координатная карта

K скалярных полей образуют **многомерный «адрес»** для каждой вершины. Если K
выбрано анатомически разумно (нос, глаза, рот, бровь), эти адреса
**различающие** — каждая точка лица имеет свой профиль.

### 6.3 Корреспонденция через nearest neighbor — это discrete functional map

Формально мы строим отображение `φ: V_B → V_A` через ближайшего соседа в
embedding-пространстве. Это **частный случай Functional Maps** где базисом
выбраны K heat-полей вместо первых K мод Laplace-Beltrami. Преимущество:
heat-поля **локализованы вокруг анкеров** и потому семантически
интерпретируемы, в отличие от глобальных eigenfunctions.

### 6.4 Cluster transfer = метка через φ

Перенос cluster label — это просто pullback функции `c: V_A → ℤ` через φ:

```
c_B(u) = c_A(φ(u))
```

Это естественная операция в category функций на мешах.

## 7. Сильные стороны

✓ **Topology-agnostic**: работает на разных triangulations  
✓ **Scale-agnostic**: после нормализации работает на разных пропорциях  
✓ **Семантически прозрачный**: каждый шаг можно интерпретировать  
✓ **Дешёвый**: только heat-diffusion (быстро) + nearest neighbor (numpy)  
✓ **Локальный**: anchor-зоны изолируют ошибки  
✓ **Не требует обучения**: чисто геометрический  
✓ **Хорошо параметризуем**: `k_nn`, `smooth_iters`, `threshold`, `time` —
   каждый имеет понятный физический смысл

## 8. Слабости и ограничения

✗ **Зависит от качества anchor'ов**: 6 случайных анкеров — плохо;
   6 анатомически разнесённых — отлично. См. `heat_corr.png` — нужно избегать
   сильно скоррелированных пар.

✗ **Симметрия лица**: левая и правая бровь имеют **идентичные heat-векторы**
   при симметричных анкерах. Решается ассиметричным расположением анкеров
   или обработкой левой/правой половины отдельно.

✗ **Disconnected components**: если на FBX глаза/зубы — отдельные геометрические
   куски, тепло до них не доходит, они выпадают из переноса. Нужна
   pre-проверка connected components.

✗ **Heat не идеальный intrinsic**: на очень разных метриках (стилизованный
   мультяшный персонаж vs реалист) тепло течёт по-разному. Резервный план:
   `voronoi` или `hks`.

## 9. Сравнение с другими режимами v3

| Аспект | heat_vec | heat_svd | **heat_align** |
|---|---|---|---|
| Гранулярность | cluster-profile | cluster-profile в SVD-basis | **per-vertex** |
| Сравнение | target vs avg-профиль | target vs avg-профиль (r-dim) | **target vs each source-vertex** |
| Граничный шум | мало (среднее сглаживает) | мало | средний (есть k-NN + mesh smooth) |
| Шумоустойчивость в profile-space | средняя | **высокая** (SVD) | средняя |
| Точность на границах | средняя | средняя | **высокая** |
| Скорость | быстро | очень быстро | средне |

**Резюме:** `heat_align` точнее на границах потому что не усредняет cluster-профиль
а сравнивает напрямую с каждой source-вершиной. Шум подавляется k-NN voting'ом
+ mesh smoothing.

## 10. Что развивать в v4

На основе этого фундамента имеет смысл:

1. **Soft assignment** — вместо hard `argmax` каждая target-вершина получает
   распределение по нескольким кластерам, delta усредняется. Швы становятся
   плавными **автоматически** без post-smoothing.

2. **Bi-directional consistency** — проверить что φ_AB и φ_BA согласованы
   (`φ_BA(φ_AB(v)) ≈ v`). Inconsistent точки — где-то ошибка корреспонденции.

3. **Heat-vector + spectral hybrid** — комбинировать K heat-полей с N
   eigenfunction'ами в одном embedding'е. Heat даёт локальную точность,
   spectral — глобальный context.

4. **Auto anchor placement** — найти K точек на меше с максимально
   взаимно-некоррелированными heat-полями (greedy selection через decorrelation).

5. **Learned anchors** — обучить (Siamese-style) heat-эмбеддинг чтобы анатомически
   одинаковые точки на разных мешах были близки в embedding'е.

6. **Cross-mesh feature transfer beyond clusters** — heat-correspondence
   универсальна, можно переносить не только motion-groups но и любые поля:
   UV, weights, vertex colors, normals.

## 11. Связь с литературой

Метод концептуально похож на:
- **Functional Maps** (Ovsjanikov et al. 2012) — но с heat-basis вместо
  Laplace eigenfunctions
- **Heat-Kernel Signature** (Sun et al. 2009) — но не как scalar dec­ript­or,
  а как многомерный vector через множество anchor'ов
- **Bi-Harmonic Distance Maps** (Lipman et al.) — heat как метрика
- **Skinning weights transfer** (Baran & Popović 2007) — переноc bone weights
  через интерполяцию, но не учитывает intrinsic geometry

Наш метод **проще** Functional Maps (не нужен оптимизатор), но **специфичнее**:
работает только когда есть осмысленные anchor-точки. Для face-rig этот сценарий
естественный — facial landmarks доступны.
